# 通用自适应规则

本文档用于沉淀一套可复用的响应式 / 自适应设计规则。它不绑定具体项目、页面或组件，适合在网站、后台、落地页和内容型页面中作为实现规范参考。

## 1. 总体原则

自适应不要只依赖单一手段，应拆成三层：

1. **断点布局**：用响应式断点处理结构变化，例如单列变双列、移动端菜单变桌面导航、移动卡片流变桌面横向布局。
2. **连续缩放**：用 `clamp()`、`min()`、`max()`、`vw`、`svh`、`rem` 等单位处理字号、间距、容器宽度和媒体尺寸的平滑变化。
3. **组件行为**：用少量运行时逻辑处理需要感知视口、滚动、触摸或交互状态的场景，例如菜单展开、轮播、滚动叙事、懒加载和动画暂停。

推荐目标是：**结构在断点处切换，尺寸在断点之间平滑变化，复杂交互在移动端降级为更稳定的阅读体验。**

## 2. 断点规则

断点应该服务于布局变化，而不是为了微调每个像素。

| 层级 | 建议宽度 | 用途 |
| :--- | :--- | :--- |
| 小屏 | `< 640px` | 手机竖屏，单列布局，弱化复杂动效 |
| 中小屏 | `>= 640px` | 大手机 / 小平板，适当增加边距和媒体尺寸 |
| 平板 | `>= 768px` | 两列内容、较大的卡片间距、更多横向空间 |
| 桌面 | `>= 1024px` | 桌面结构切换，首屏图文分栏，sticky 区域可启用 |
| 大桌面 | `>= 1280px` | 多列网格、完整横向导航、宽内容区 |
| 超宽屏 | `>= 1440px` | 控制整体放大节奏，避免内容无限拉宽 |

规则：

- 移动端优先写基础样式，再逐层增加 `md`、`lg`、`xl` 等增强样式。
- 断点只处理明显结构变化，常规字号和间距优先用连续缩放。
- 如果桌面导航或复杂工具栏空间需求较高，可以使用高于普通桌面的自定义断点，例如 `>= 1400px`。
- 不要默认认为 `1024px` 就一定可以承载完整桌面导航，真实断点应由内容宽度决定。

## 3. 设计 Token

建议把跨页面复用的响应式尺寸抽象成 CSS 变量，而不是在组件里重复写死数值。

常用 token 类型：

```css
:root {
  --shell-sm: clamp(1.25rem, 5vw, 4.5rem);
  --shell-md: clamp(1.5rem, 8vw, 7rem);
  --shell-lg: clamp(1.75rem, 12vw, 11rem);

  --header-height: clamp(3.75rem, calc(1.75rem + 2vw), 4.5rem);

  --type-hero: clamp(3rem, calc(2.5rem + 2vw), 6rem);
  --type-title: clamp(2rem, 4vw, 4rem);
  --type-body: clamp(1rem, 1.2vw, 1.15rem);

  --media-stage-height: clamp(24rem, 34vw, 40rem);
  --carousel-stage-height: clamp(32rem, 42vw, 41rem);
}
```

规则：

- 页面左右边距统一从 shell token 获取。
- 标题、正文、卡片标题分别使用不同字号 token。
- 媒体区域和轮播区域使用稳定高度或宽高比，避免图片加载后布局跳动。
- token 名称表达用途，不表达具体页面。

## 4. 大屏缩放

超宽屏不能简单让内容无限变宽。推荐两种方式：

1. **内容宽度封顶**：正文、卡片网格、表单等设置 `max-width`。
2. **有限整体放大**：超过某个桌面基准后，让 `rem` 或关键 token 以受控比例放大。

示例：

```css
@media (min-width: 90rem) {
  html {
    font-size: calc(100vw / 90);
  }
}
```

使用这类规则时要注意：

- 所有 `rem` 尺寸都会受影响。
- 新增组件时要确认它在 1440px、1920px、超宽屏下是否仍然合理。
- 需要给内容宽度和图片宽度设置上限，否则大屏会显得松散。

## 5. 页面壳层

页面壳层负责统一左右安全边距和最大内容宽度。

推荐写法：

```css
.page-shell {
  width: 100%;
  margin-inline: auto;
  padding-inline: var(--shell-md);
}

.content-shell {
  width: 100%;
  max-width: 90rem;
  margin-inline: auto;
  padding-inline: var(--shell-sm);
}
```

规则：

- 同一类页面使用同一组 shell，不要每个区块单独写边距。
- 需要全出血图片或背景时，让背景满宽，内容仍放回 shell 内。
- 横向滚动通常是错误信号，应优先检查 `w-screen`、绝对定位、未设置 `min-width: 0` 的 flex/grid 子项。

## 6. 首屏规则

首屏应同时处理高度、标题、媒体和移动端浏览器工具栏。

推荐规则：

- 外层使用 `min-height: 100svh`，必要时配合 `height: 100svh`。
- 文字区宽度要受控，避免长标题在桌面端横向溢出。
- 大标题使用 `clamp()` 或全局字号 token。
- 移动端图片可以放在标题下方，桌面端图片可以改为绝对定位或分栏。
- 人物、产品、海报等关键图像应设置明确的 `object-fit` 和最大尺寸。
- 首屏图像优先加载，非首屏图像延迟加载。

移动 / 桌面两套媒体位置示例：

```html
<div class="hero-media-mobile lg:hidden">...</div>
<div class="hero-media-desktop hidden lg:flex">...</div>
```

## 7. 导航规则

导航自适应的核心不是屏幕宽度，而是内容是否放得下。

规则：

- 小屏使用菜单按钮，展开后纵向排列导航项。
- 大屏显示完整导航、语言切换、社交入口和外链按钮。
- logo 使用 `max-width: min(...vw, ...rem)`，避免挤压按钮。
- 固定导航需要设置稳定高度 token。
- 首屏透明导航可在滚动后切换为半透明背景，但要保证可读性。
- 如果导航项较多，完整导航断点可以高于普通桌面断点。

## 8. 卡片和网格

卡片和网格是最常见的自适应区域。

规则：

- 手机端默认单列。
- 平板端按内容密度切换为两列。
- 桌面端最多三到四列，避免每列过窄。
- 卡片图片使用固定宽高比，例如 `aspect-ratio: 16 / 10`。
- 卡片标题和摘要要允许换行，必要时设置最小高度或统一行高。
- grid/flex 子项加 `min-width: 0`，防止长文本撑破布局。
- 不要靠固定像素高度硬压文本，优先让内容自然撑开。

示例：

```html
<div class="grid grid-cols-1 gap-6 md:grid-cols-2 xl:grid-cols-3">
  ...
</div>
```

## 9. 轮播规则

轮播需要特别控制舞台高度、卡片尺寸和移动端体验。

规则：

- 轮播舞台使用稳定高度 token，避免切换时页面跳动。
- 中心卡片和两侧卡片可通过 `scale`、`opacity`、`translateX` 做层级差异。
- 移动端优先使用单卡片、普通滑动或纵向列表，不强制保留复杂 3D 效果。
- 图片内容多的海报优先使用 `object-fit: contain`，视觉氛围图可用 `cover`。
- 控制按钮和分页点应随轮播整体缩放，而不是脱离舞台单独定位。

## 10. 滚动叙事规则

桌面端可以使用 sticky / scroll-driven 叙事，但移动端应降级。

桌面端规则：

- 每个内容项占一个明确滚动段，例如 `100vh`。
- 左侧文案和右侧图片都使用 sticky，保持阅读焦点稳定。
- 图片列宽度用 `clamp()` 控制，避免太小或太大。
- 滚动监听应节流到 `requestAnimationFrame`。
- 离开视口后暂停滚动计算和动画。

移动端规则：

- 不使用复杂 sticky。
- 改为普通卡片流。
- 每个卡片包含完整图片、标题、说明和链接。

## 11. 文字规则

文字自适应要同时考虑中英文、长词、标题和按钮。

规则：

- 标题使用 `clamp()` 控制字号。
- 正文使用稳定行高，通常 `1.6` 到 `1.8`。
- 长标题设置 `max-width`，必要时使用 `text-wrap: balance` 或允许断词。
- 按钮文字不能依赖缩放兜底，最长语言版本也要能放下。
- 中英文不要强行共用所有字号，必要时用语言分支微调。
- 不要用 `vw` 直接控制所有字号，容易在极窄或极宽屏失控。

## 12. 媒体规则

图片、视频和图标要明确尺寸策略。

规则：

- 图片容器先有尺寸，再加载图片。
- 卡片图片使用 `aspect-ratio`。
- 首屏关键图片 eager，非首屏 lazy。
- 使用 `object-fit: cover` 时确认裁切可接受。
- 信息型海报、截图、证书类图片通常使用 `contain`。
- 图标按钮使用固定宽高，避免文字或图标加载后改变布局。

## 13. 移动端缩放

是否允许用户缩放应由产品场景决定。

如果目标是强设计一致性，可以锁定 viewport：

```ts
export const viewport = {
  width: "device-width",
  initialScale: 1,
  minimumScale: 1,
  maximumScale: 1,
  userScalable: false,
  viewportFit: "cover",
};
```

但要注意：

- 锁定缩放后，移动端排版必须自己处理好可读性。
- 表单、按钮、链接的触控面积要足够大。
- 不应把“用户可以放大”当作布局兜底。

## 14. 动效规则

响应式不仅是布局，也包括性能。

规则：

- 首屏动画要轻，避免阻塞主要内容。
- 非首屏动画进入视口后再启动。
- 页面隐藏时暂停动画、轮播和定时器。
- 尊重 `prefers-reduced-motion: reduce`。
- 移动端降低粒子、WebGL、阴影和大面积 blur 的强度。

示例：

```css
@media (prefers-reduced-motion: reduce) {
  * {
    animation: none !important;
    transition-duration: 0.01ms !important;
  }
}
```

## 15. 实现检查清单

上线前至少检查以下视口：

- 375px 手机竖屏。
- 390px 手机竖屏。
- 768px 平板。
- 1024px 平板横屏 / 小桌面。
- 1440px 桌面基准。
- 1920px 大桌面。

检查项：

- 页面没有横向滚动。
- 标题、按钮、卡片文字没有溢出。
- 导航在每个宽度下都能正常操作。
- 首屏高度没有被移动浏览器地址栏挤坏。
- 图片裁切符合内容语义。
- 网格列数和卡片宽度合理。
- sticky、轮播、动画在移动端有降级方案。
- 大屏内容没有被无限拉宽。

## 16. 修改优先级

调整自适应时按这个顺序处理：

1. 先判断是结构问题还是尺寸问题。
2. 结构问题用断点解决。
3. 尺寸问题优先改 token 或 `clamp()`。
4. 单个组件特殊问题再加局部样式。
5. 如果移动端体验复杂，优先降级成简单布局。
6. 最后再做动画和细节微调。

核心原则：**能抽象成 token 的不要散落在组件里，能用连续缩放的不要堆断点，能在移动端简化的不要强行复刻桌面复杂交互。**
